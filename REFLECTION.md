## 一、Superpowers 技能：哪些发挥最大作用，哪些"形式大于实质"

### 发挥最大作用的技能

**1. `brainstorming`（设计驱动）—— 9/10**

Brainstorming 是本次项目中发挥作用最大的技能。它的价值不在于"帮我想出好点子"，而在于**强制我在写代码之前把设计写清楚**。分节确认机制（架构 → 护栏 → 五维度 → WebUI → 凭据 → 分发）让每一个设计决策都有明确的签字节点。如果没有这个流程，我大概率会直接跳到"写一个 agent 主循环"然后边写边想，最终产出一个拼凑的产物。

但 brainstorming 有一个盲区：它不会主动对照需求文件检查完整性。我在第 1 轮迭代中打断智能体，要求重读三份要求文件，才发现遗漏了 WebUI、冷启动验证、机制演示等关键要求。这说明 brainstorming 在"探索性设计"方面很强，在"合规性检查"方面需要外部触发。

**2. `subagent-driven-development`（任务派发）—— 8/10**

21 个 Task 全部派发 fresh subagent，每个 subagent 只拿到一个独立任务，不继承历史上下文。这种模式的优势在于：
- 每个 subagent 的上下文干净，不会被之前任务的细节干扰
- 如果某个 subagent 出错，不影响其他 task
- 并行派发（Task 18-21）安全无冲突

代价是每个 task 都需要重新理解项目背景，对于有依赖关系的 task（如 Task 14 依赖 10+ 个模块），subagent 需要在 dispatch prompt 中获得足够的上下文信息。

**3. `test-driven-development`（TDD 强制）—— 7/10**

TDD 在本次项目中的价值不在于"先写测试能发现 bug"（因为 PLAN 中的代码已经写好），而在于**它让每个 subagent 的输出有一个明确的验收标准**。每个 task 的最后一个步骤都是"运行测试确认通过"，这让 subagent 知道自己是否完成了任务。

### "形式大于实质"的技能

**`using-git-worktrees`** — 在本项目中基本没有发挥作用。我们在 main 分支上完成了所有 21 个 task，没有使用 worktree 隔离。原因是：
1. 项目规模不大，21 个 task 之间没有真正需要隔离的冲突
2. worktree 的 overhead（每个 worktree 需要独立的环境配置）在单机开发中收益不大
3. 时间紧迫，直接在主分支上推进更高效

但 worktree 在多人协作或多 feature 并行开发中确实有价值。只是在本项目的场景下，它的投入产出比不高。

## 二、TDD 在 AI 协作下：阻碍还是放大器？

**我的结论：TDD 在 AI 协作下是放大器，但有前提条件。**

前提条件是：**测试代码必须已经写好（在 PLAN 中），subagent 的工作是"让测试变绿"而非"先想测试再写代码"。**

在本次项目中，每个 Task 的 PLAN 都包含了完整的测试代码和实现代码。Subagent 的工作流程是：
1. 创建测试文件 → 运行 → 失败（红）
2. 创建实现文件 → 运行 → 通过（绿）

这个流程的价值在于：
- 给 subagent 一个明确的"完成"信号（测试通过 = 完成）
- 防止 subagent 过度实现（YAGNI — 只写让测试通过的代码）
- 让 Controller（我）可以验证 subagent 的输出质量

如果换成"让 subagent 自己设计测试 + 实现"，TDD 可能会变成阻碍。因为 subagent 可能写出覆盖不全的测试，或者花大量时间在测试设计上偏离核心任务。

**一个反面案例：** Task 20（机制演示）的 subagent 在运行 test_demo_2 时遇到了 `_handle_run_tests` 委托给 `_handle_execute_shell` 导致 tool_name 不一致的问题。如果当时没有预先写好的测试，subagent 可能不会发现这个设计缺陷。

## 三、Subagent-Driven 工作流的自主运行能力

**Subagent 能在不偏离主题的情况下自主运行 1 个 Task（约 2-5 分钟）。**

21 个 Task 中，没有一个 subagent 报告偏离主题。原因：
1. 每个 Task 的 PLAN 代码足够完整，subagent 是"转录"而非"设计"
2. Dispatch prompt 中包含了明确的文件路径、代码内容、验证步骤
3. 每个 task 颗粒度小（1-2 个文件），不涉及跨模块设计决策

**但如果让 subagent 自主运行多个 Task，偏离概率会显著增加。** 因为 subagent 在完成 Task N 后，没有 Task N+1 的完整上下文，必须依赖 Controller 提供下一个 dispatch。

**最有效的 Task 颗粒度：** 1-2 个文件，2-5 分钟完成。这种颗粒度下：
- Subagent 的上下文窗口充足
- 验证步骤简单明确
- 失败后重试成本低

## 四、SPEC/PLAN 质量对实现质量的影响

**核心发现：PLAN 中代码的完整度直接决定实现质量。**

本次项目中，PLAN 的每个 Task 都包含了完整的测试代码和实现代码。Subagent 的工作本质上是"转录代码 + 运行验证"。这使得 21 个 Task 全部一次通过。

**如果 PLAN 中只有文字描述而没有代码，** subagent 需要自己设计实现，此时：
- 不同 subagent 的代码风格会不一致
- 模块间接口可能出现不匹配
- 测试覆盖率无法保证

**一个具体案例：** Task 6（工具执行器）的 subagent 发现 ConfigLoader 需要在 `__init__` 中初始化 `_config` 为 `DEFAULT_CONFIG.copy()`。如果 PLAN 中没有提供完整的代码，subagent 可能不会考虑这个边界情况，后续 Task 在使用 ConfigLoader 时就会出现空配置错误。

**结论：** PLAN 中的代码越完整，实现质量越高，偏离越少。这意味着 PLAN 的编写者（我）需要投入更多精力在"写清楚"上，而不是依赖 subagent 的"理解能力"。

## 五、最有效的 Prompt / Context 策略

**策略 1：代码即规约（最有效）**

Dispatch prompt 中直接提供完整的测试代码和实现代码，而非用自然语言描述需求。这消除了 subagent 对需求的理解偏差。

**策略 2：明确的验证步骤**

每个 Task 的 prompt 末尾都有明确的验证命令：
```
pytest tests/test_xxx.py -v
```
这让 subagent 知道"完成"的标准是什么。

**策略 3：最小化上下文**

每个 dispatch prompt 只包含：
- 当前 Task 的完整代码
- 依赖的模块接口（而非完整实现）
- 全局约束
- 验证步骤

不传递历史会话、不传递其他 Task 的细节。这避免了上下文污染。

**策略 4：文件路径明确**

所有文件路径使用绝对路径（`D:\homework in summar\src\pyharness\xxx.py`），避免 subagent 在错误的目录下创建文件。

## 六、凭据与分发：让你想清楚了哪些问题

### 凭据安全

凭据安全的要求迫使我思考了三个问题：
1. **Key 的生命周期：** 从录入 → 存储 → 使用 → 清除，每个环节都需要安全设计
2. **威胁模型：** 不只是"不要提交到 Git"，还包括日志泄露、进程环境可见、shell history
3. **跨平台：** OS keyring 在不同平台（Windows/macOS/Linux）的 API 完全不同，需要统一的抽象层

这些是纯"写代码"时不会考虑的问题，但它们是"一个可靠的系统"必须回答的工程问题。

### 分发

Docker 分发的要求迫使我思考：
1. **依赖管理：** 容器内的 Python 版本、系统库、pip 包需要精确控制
2. **Key 如何在容器内配置：** 容器内的 keyring 可能不可用，需要挂载加密文件或通过 WebUI 配置
3. **端口暴露：** 8080 端口需要在 Dockerfile 中显式声明

这些问题是"在别人机器上跑起来"必须解决的，而不仅仅是"在我机器上能跑"。

## 七、如果重做，我会改变什么

1. **先做冷启动验证再做实现：** 本次是先实现后写文档，但正确的顺序应该是冷启动验证 → 修正 SPEC/PLAN → 实现。冷启动验证能提前暴露 SPEC 缺陷，避免实现时的返工。

2. **使用 git worktree：** 虽然在本项目中 worktree 的收益不大，但作为一个完整的工程流程，应该在每个 feature 开始时创建 worktree，保持 main 分支干净。

3. **更早引入验证脚本：** 验证脚本（verify.bat/verify.sh）应该和第一个 Task 一起创建，每完成一个 Task 就运行一次，而不是等到全部完成后再补。

4. **在 dispatch prompt 中增加更多接口信息：** 对于依赖多个模块的 Task（如 Task 14），应该在 prompt 中更详细地说明每个依赖模块的接口签名，减少 subagent 的猜测。

## 八、对 Superpowers 方法论的批判

### 它假设了什么

1. **假设 SPEC 是完整且正确的：** Superpowers 的整个流程（brainstorming → plan → implement）建立在一个假设上：brainstorming 产出的 SPEC 足够完整，subagent 可以按图索骥。但在实际中，SPEC 的完整性很难在第一次就达到。

2. **假设 subagent 能理解 PLAN 中的代码：** 本次项目中 PLAN 包含了完整代码，subagent 的工作是转录。但如果 PLAN 中只有文字描述，subagent 的理解质量会显著下降。

3. **假设 TDD 能在 AI 协作中自然运作：** TDD 的前提是"知道要测什么"，但 AI 产出的测试可能覆盖不全或测试了错误的东西。

4. **假设任务可以独立拆分：** 不是所有任务都能拆成 2-5 分钟、1-2 个文件的颗粒度。某些集成任务（如 Agent 主循环）天然需要跨多个模块的协调。

### 这些假设在本项目中成立吗

| 假设 | 成立程度 | 说明 |
|------|---------|------|
| SPEC 完整正确 | 部分成立 | 经用户多次打断修正后基本完整，但冷启动验证尚未进行 |
| Subagent 理解 PLAN | 成立 | 因为 PLAN 包含了完整代码 |
| TDD 自然运作 | 成立 | 因为测试代码已写在 PLAN 中 |
| 任务可独立拆分 | 大部分成立 | 21 个 Task 中 19 个独立，Task 14 是集成任务 |

### 最大的不足

Superpowers 流程中缺少一个**显式的"需求合规检查"环节**。Brainstorming 结束后，没有机制自动检查 SPEC 是否覆盖了需求文件中的所有要求。这个检查目前完全依赖用户主动提出，容易遗漏。

**建议：** 在 brainstorming 和 writing-plans 之间增加一个"需求对照检查"步骤，由智能体主动逐条对照需求文件，标注已覆盖和未覆盖的项。

## 九、总结

本次项目让我对"当 AI 能完成大部分编码工作时，一个工程师的真正价值在哪里"有了第一手的答案：

**工程师的价值不在于写代码（AI 可以写），而在于：**
1. **定义"做什么"** — SPEC 和 PLAN 的质量直接决定 AI 的输出质量
2. **检查"做对了吗"** — 冷启动验证、代码评审、合规检查这些环节 AI 不会自发做
3. **想清楚"为什么"** — 凭据安全、分发、架构选型这些工程决策需要人类判断

Superpowers 是流程脚手架，它守住了 TDD、评审、计划这些容易松懈的纪律，但它不能替你回答"做什么"和"做对了吗"。这两个问题，正是本次项目最核心的训练。
