## 二、Brainstorming 关键节点

### 节点 1：需求重读（第 1 轮关键迭代）

**背景：** 在选定项目类型 A（Coding Agent Harness）和 Python 技术栈后，智能体开始按部就班提问。此时用户打断，要求重新仔细阅读全部要求文件。

**智能体的追问：** 无 — 用户主动要求重读。

**发现的问题：** 智能体在初次阅读时遗漏了以下关键要求：
- §五.9：必须提供 WebUI 接口和线上部署 URL
- §五.6：CI 配置是 `.gitlab-ci.yml`（与 §4.8 的 "GitHub Actions" 矛盾）
- §4.5：冷启动验证 — 必须用不同类型的 agent 验证 SPEC+PLAN
- A.6：机制演示 — 3 个确定性场景
- A.4-D：六个维度都要有最低实现，但选一个深入

**处理决策：** 要求智能体重读三份文件，逐条对照检查，确保不遗漏。

**反思：** 这是整个 brainstorming 过程中最关键的节点。智能体在"快速推进"的惯性下跳过了细节，用户的主动介入避免了后续大量返工。这印证了课程要求的核心观点：**智能体不能替你回答"做什么"和"做对了吗"。**

---

### 节点 2：架构方案选择（第 2 轮关键迭代）

**智能体的提问：** "提出 3 个架构方案：管道式、事件驱动、状态机。推荐管道式。"

**三个方案：**

| 方案 | 核心思想 | 优点 | 缺点 |
|------|---------|------|------|
| 管道式 | 线性管线，每环节独立 | 职责清晰，护栏天然可测 | 不够灵活 |
| 事件驱动 | 事件总线，松耦合 | 扩展性好 | 调试困难，over-engineering |
| 状态机 | 有限状态机，护栏是守卫条件 | HITL 天然建模 | 状态组合爆炸 |

**智能体的推荐：** 管道式 — "护栏天然独立可测，完美满足 §A.4-C 的判定标准"

**用户决策：** 采纳推荐，选择管道式。

**为什么采纳：** 智能体的推荐理由与项目核心要求（机制必须可 mock 单测）直接对齐，判断准确。

---

### 节点 3：WebUI 设计迭代（第 3 轮关键迭代）

**智能体的初版设计：** 单栏终端式界面，类终端渲染，审批请求内联显示。

```
$ 你: 帮我写一个计算器
[agent] 正在分析需求...
[agent] 创建 calculator.py
⚠️ 护栏拦截: 命令 "rm -rf /" 被规则护栏拦截
$ █
```

**用户的反馈：** "这个 UI 界面可以更加优化一下"

**智能体的修订：** 三栏式布局 — 左栏文件树 + 中栏终端主区域 + 右栏状态面板 + 底栏操作区。

**改进点：**
- 增加了文件树面板（实时反映文件变更）
- 增加了状态面板（轮次、token、护栏拦截数）
- 增加了底栏（配置、护栏日志、停止按钮）
- 增加了首次访问的 API Key 配置弹窗

**处理决策：** 用户推翻初版设计，要求优化。智能体提出的三栏式方案被采纳。

**反思：** 智能体的初版设计过于简单，用户作为"产品经理"提出了合理的 UX 要求。这说明在设计阶段，人类对"什么体验是好的"仍然有不可替代的判断力。

---

### 节点 4：五个维度检查（第 4 轮关键迭代）

**用户的提问：** "先查看是否满足我们的作业的要求"

**智能体的检查：** 逐条对照两份要求文件，列出了已满足项和缺失项：
- 已满足：Agent 主循环、Mock LLM、不寄生框架、六维度设计、四类机制等
- 缺失：WebUI 设计、凭据安全存储、停机判断机制、ActionParser 设计、Open Design 选择、Docker 分发设计、CI 方案

**处理决策：** 用户确认五维度没问题，继续逐项设计缺失部分。

**反思：** 这是用户主动要求的"合规检查"，而非智能体自发的。智能体在 brainstorming 中倾向于"向前推进"而非"向后检查"，需要外部触发才能停下来对照需求。

---

### 节点 5：凭据安全设计（第 5 轮关键迭代）

**背景：** 智能体用英文呈现了凭据安全设计，用户要求用中文重新说一遍。

**智能体的设计：** OS keyring 优先 + 加密文件备选，威胁模型、存储方案、用户操作流程。

**用户决策：** 内容上无异议，仅要求语言切换。

**为什么采纳：** 设计覆盖了威胁模型（文件系统、Git、日志、进程环境）、存储方案（keyring 优先 + 加密文件备选）、用户流程（CLI 和 WebUI 双入口），符合 §3.1 的全部要求。

---

### 节点 6：项目命名与目录结构

**智能体的提议：** 项目名称 "PyHarness"，目录结构按模块划分（models/guardrail/tools/feedback/memory/context/loop/webui/credential）。

**用户决策：** 直接采纳。

**为什么采纳：** 命名简洁直观，目录划分与 SPEC 中的模块设计一一对应，每个模块有独立的测试文件。

---

### 节点 7：SPEC 完整性检查

**智能体的行为：** 写完 SPEC.md 后，主动进行自检：
- 检查 10 个必含章节是否全覆盖
- 检查 A.5 领域与机制设计是否包含
- 检查用户故事数量（7 个 ≥ 5）
- 检查数据模型与功能模块的一致性

**用户决策：** SPEC 通过，进入 PLAN 生成。

## 四、Brainstorming 反思

### 做得好的地方

1. **分节确认机制：** 架构 → 护栏 → 五维度 → WebUI → 凭据 → 分发 → 命名，每节独立确认，节奏合理。用户有充分的时间在每个节点介入和修正。

2. **多方案比较：** 在架构选择阶段提出了 3 个方案并给出推荐理由，而非直接给一个答案。这符合 brainstorming 的"explore alternatives"原则。

3. **自检机制：** 写完 SPEC 后主动对照要求文件逐条检查，发现了遗漏项并补充。

4. **技术选型有理由：** 每个选择都附带了"为什么"，如 Python 选型理由（LLM 生态、keyring 跨平台、pytest 成熟），而非空泛的"Python 好用"。

### 做得不好的地方

1. **初次阅读不仔细：** 在用户打断之前，智能体遗漏了 WebUI、冷启动验证、机制演示等关键要求。说明智能体在"快速推进"模式下有遗漏细节的倾向。这需要用户在关键节点主动介入检查。

2. **合规检查不主动：** 智能体不会自发地对照需求文件检查设计完整性，需要用户明确提出"检查是否满足要求"。

3. **UI 设计初版过于简陋：** 智能体对 UI 的初始设计偏向"功能可用"而非"体验良好"，需要用户 push 才能产出更好的设计。

4. **语言切换不敏感：** 在用户明确要求中文交流后，智能体在凭据安全设计部分仍然使用了英文，需要用户再次提醒。

### 对 Superpowers brainstorming 技能的批判

Brainstorming 技能的设计假设是"智能体会主动追问并逐步澄清需求"。这个假设在本项目中**部分成立**：
- 成立的部分：智能体确实会追问技术选型、架构方案、设计细节
- 不成立的部分：智能体不会主动检查"是否遗漏了需求文件中的要求"，需要外部触发

这意味着 brainstorming 技能在"探索性设计"方面表现良好，但在"合规性检查"方面存在盲区。**建议在 brainstorming 流程中增加一个显式的"需求对照检查"环节**，而非依赖用户主动提出。
